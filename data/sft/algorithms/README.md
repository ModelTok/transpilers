# Algorithm SFT pairs (issue #57)

Toolchain-free SFT pairs generated by
[`scripts/sft/build_algorithm_pairs.py`](../../../scripts/sft/build_algorithm_pairs.py)
by running the transpiler's **algorithmic** path (no model, no network, no
external compiler) over small self-contained function corpora.

Regenerate:

```bash
uv run python scripts/sft/build_algorithm_pairs.py
```

## Directions

| file | direction | issue ask |
|------|-----------|-----------|
| `cpp_mojo_pairs.jsonl`    | C++ → Mojo    | yes (headline) |
| `c_mojo_pairs.jsonl`      | C → Mojo      | yes (headline) |
| `cpp_python_pairs.jsonl`  | C++ → Python  | yes |
| `python_mojo_pairs.jsonl` | Python → Mojo | bonus (repo also trains on this) |
| `python_rust_pairs.jsonl` | Python → Rust | bonus, behaviorally verified in-env |
| `python_zig_pairs.jsonl`  | Python → Zig  | bonus |
| `python_c_pairs.jsonl`    | Python → C    | bonus |

`rust → mojo` is in the issue but the transpiler has **no Rust frontend**, so it
is out of scope here.

## Record schema

Matches the other SFT corpora (`instruction` / `input` / `output`) plus
provenance:

```json
{
  "instruction": "Translate the following C++ function to idiomatic Mojo. ...",
  "input": "int gcd(int a, int b) { ... }",
  "output": "```mojo\ndef gcd(var a: Int, var b: Int) -> Int: ...\n```",
  "source_lang": "cpp", "target": "mojo", "func": "gcd",
  "status": "unverified_no_runner", "verified": false,
  "verify_note": "no in-env runner for cpp->mojo"
}
```

## Verification status (3-way, honest)

- **`verified`** — behavioral match over generated inputs. Only `python` source
  with a `python`/`rust` target is runnable in this pure-Python environment, so
  only those can reach `verified` here.
- **`unverified_no_runner`** — emitted but no in-env oracle/runner for this
  direction (all `→mojo`/`→zig`/`→c` and all `cpp`/`c` source). The
  compile/behaviour gate is **deferred** to a box with `g++`/`mojo`/`gcov`; the
  pair is *not* claimed correct. The headline `→Mojo` pairs live here.
- **`failed`** — the harness *ran* and the target diverged from the source
  oracle: a demonstrably wrong translation. These are split into
  `*.rejects.jsonl` and **excluded from the training files** so a known-wrong
  pair never enters SFT (same keep-only-trustworthy discipline as
  `build_cpp_mojo_dataset.py`).

### Known divergence class surfaced

The Python→Rust rejects (`gcd`, `lcm`, `fast_pow`) diverge on negative operands:
Python's **floored** `%`/`//` vs Rust/C's **truncated** `%`/`/`
(`-1 % 2` → Python `1`, Rust/C `-1`). The unverified `→mojo`/`→zig`/`→c` pairs
for those same functions almost certainly share this bug — "no_runner" means
*not yet checked*, **not** "presumed correct". A dedicated floored-mod emission
fix is tracked separately.
